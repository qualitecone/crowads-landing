CROWADS Engineering Spec

Complete product specification for the CROWADS pooled performance marketing platform. Not an MVP scope. Designed to be handed to a lead developer and shipped in 8 months by a 4-engineer team.

Version 1.0· 21 June 2026· Owner: Ale & Gio (Ale & Gio)· Confidential
Reading note. This document describes the FINAL product, not an MVP. Every feature listed must be implemented, tested and in production before the commercial go-live. Total complexity: ~3,500 engineering hours, equivalent to 4 senior developers × 5 months, or 2 senior developers × 10 months.

3.1 Sample schema — campaigns

CREATE TABLE campaigns (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  startup_id UUID NOT NULL REFERENCES startup_profiles(user_id),
  slug TEXT UNIQUE NOT NULL,
  title TEXT NOT NULL,
  description TEXT NOT NULL,
  vertical TEXT NOT NULL,
  geo_targeting TEXT[] NOT NULL,

  budget_target_eur DECIMAL(12,2) NOT NULL,
  budget_min_eur DECIMAL(12,2) NOT NULL,
  budget_raised_eur DECIMAL(14,2) DEFAULT 0,

  revenue_share_pct DECIMAL(5,2) NOT NULL,
  attribution_window_days INT NOT NULL DEFAULT 90,
  campaign_duration_days INT NOT NULL DEFAULT 90,
  revenue_share_duration_months INT NOT NULL DEFAULT 12,

  expected_roas DECIMAL(5,2),
  expected_cac_eur DECIMAL(10,2),
  expected_investor_premium_pct DECIMAL(5,2),

  ad_platforms TEXT[] NOT NULL,
  brand_assets_url TEXT,
  buyer_persona JSONB,

  setup_fee_eur DECIMAL(10,2) NOT NULL,
  management_fee_pct DECIMAL(5,2) NOT NULL DEFAULT 7.0,
  performance_bonus_threshold_roas DECIMAL(5,2) DEFAULT 3.0,
  performance_bonus_pct DECIMAL(5,2) DEFAULT 50.0,

  status TEXT NOT NULL DEFAULT 'draft' CHECK (status IN (
    'draft', 'pending_review', 'open_for_funding', 'fully_funded',
    'awaiting_ads_launch', 'ads_running', 'ads_completed',
    'attribution_phase', 'completed', 'cancelled', 'failed'
  )),

  funding_open_at TIMESTAMPTZ,
  funding_close_at TIMESTAMPTZ,
  ads_launch_at TIMESTAMPTZ,
  ads_end_at TIMESTAMPTZ,
  attribution_end_at TIMESTAMPTZ,

  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_campaigns_status ON campaigns(status);
CREATE INDEX idx_campaigns_startup ON campaigns(startup_id);
CREATE INDEX idx_campaigns_vertical ON campaigns(vertical);

Section 9

Tracking & attribution

9.1 Click → conversion flow

1. User sees ad on Meta (Reel, Story, Feed) 2. Click → https://track.crowads.com/c/{tracker_id} 3. tracking-service logs click event to ClickHouse 4. 302 redirect to startup URL with appended UTMs: https://startup.com/landing?utm_source=crowads &utm_campaign={campaign_id}&utm_content={ad_id} &cw_click={click_id} 5. User browses → CROWADS pixel fires + localStorage cw_click_id 6. User completes purchase 7. Shopify webhook → CROWADS receives order 8. Match by UTM + click_id + attribution window 9. INSERT into revenue_attributions 10. Real-time campaign metrics update

9.2 Attribution models

  • Default: Last-paid-click, 30-day window
  • Configurable per campaign: 7d / 14d / 60d / 90d
  • Options: First-click, linear multi-touch, position-based, data-driven (requires 100+ historical conversions)

9.3 De-duplication

Unique key (source, external_order_id) prevents double counting. Each click_id generates at most one attributed conversion. Refunds within 14 days subtract from attributed revenue.

9.4 Privacy & cookieless tracking

  • First-party cookies on .crowads.com or CNAMEd subdomain
  • Server-side conversion API fallback (Meta CAPI, Google Enhanced Conversions)
  • LocalStorage + lite fingerprinting (browser + lang + timezone, no PII)
  • GDPR-compliant consent banner integrated

9.5 Fraud detection

IP velocity check, click→conversion time anomaly (under 5s = suspicious), user-agent fingerprint mismatch, geo mismatch, Cloudflare bot management, ML anomaly score.

Section 10

Escrow & payments

10.1 Architecture

Dedicated "Escrow Pool" bank account (regulated EU/UAE bank), separated from operating funds. Internal ledger on PostgreSQL with balance_after_eur field on every transaction. Daily auto-reconciliation between bank balance (via Open Banking API) and internal ledger. Discrepancy → immediate alert + freeze.

10.2 Investor inbound flow

1. Investor confirms €1,000 subscription to Campaign X 2. CROWADS generates Stripe Checkout OR shows IBAN + unique reference 3. User pays 4. Stripe webhook charge.succeeded → subscription = 'paid' 5. Virtual ledger entry: escrow_transactions ('deposit') 6. Email confirmation 7. When campaign 'fully_funded' → funds 'committed' 8. When ads launch → progressive outflow to ad spend

10.3 Monthly distributions

Cron on day 20 of every month: 1. For each campaign in attribution_phase: Calculate revenue_share_payment of the month 2. For each paid revenue_share_payment: Calculate pro-rata distributions per investor 3. Batch SEPA Credit Transfer for amounts > €50 4. Wallet accumulation for amounts < €50 5. Email notifications to investors 6. audit_log entries

10.4 Invoicing

Investor pays → CROWADS issues invoice "Performance Marketing Services - Campaign X". Startup pays revenue share → CROWADS issues invoice "Revenue Share - Campaign X". Distribution to investor → CROWADS issues credit note. SDI (eFattura) integration for Italian clients.

Section 11

Auth, KYC, AML

11.1 Auth stack

  • JWT access token 15 min, refresh 30 days, rotation enabled
  • 2FA TOTP (Google Authenticator) mandatory for balance > €10k
  • Magic link login optional
  • FIDO2 hardware key supported for admin

11.2 KYC flow (Sumsub)

User uploads document → Sumsub OCR + liveness check → sanctions screening (PEP, OFAC, EU lists) → webhook to CROWADS → approved/rejected. Average: 24-72 hours, cost €1.5/verification.

11.3 AML & ongoing monitoring

Automatic flag for cumulative > €10,000 over 12 months. Source of funds verification for > €5,000. Weekly sanctions list re-screening. Pattern detection (structuring). SAR generation when triggered. Outsourced compliance officer (monthly retainer).

Section 16

Auto-Pilot algorithm

16.1 Scoring formula (composite score per campaign)

score = w1 * historical_roas_vertical
      + w2 * startup_tier_score          (bronze=1, silver=2, gold=3, platinum=4)
      + w3 * startup_track_record        (% paid on time on previous campaigns)
      + w4 * remaining_budget_ratio      (favor near-funded for fast close)
      - p1 * risk_penalty                (per high-risk if profile is low)

Weights tuned via ML model retrained monthly on historical outcomes.

16.2 Allocation algorithm

  1. Filter open campaigns by preferred vertical + risk-profile-compatible
  2. Calculate score
  3. Allocate with constraints: max % per campaign, min # of campaigns
  4. Quadratic programming to optimize expected ROI / variance

16.3 Rebalancing

Monthly or on-trigger (new top-rated campaign, end of campaign). Released capital (distributions) automatically reallocated. Committed capital is never reallocated (legal constraints).

Section 18

Compliance & audit logging

18.1 Audit log requirements

Every critical action logged in audit_log: user actions (login, signup, KYC, subscription, withdrawal), admin actions (approve/reject, manual override, refund), system actions (distributions, payments, attribution updates), integration events (webhooks).

Section 21

Security requirements

21.1 Application security

OWASP Top 10 mitigations. CSP, HSTS, X-Frame-Options. Input validation via Zod. SQL injection prevention via Prisma. CSRF tokens. Rate limiting. Cloudflare bot management.

21.2 Auth security

Argon2id password hash, min 12 chars, HaveIBeenPwned breach check. Mandatory 2FA for balance > €10k. Session timeout 30min idle, max 30 days. Login notifications on new device.

21.3 Data security

Encryption at-rest (AES-256). TLS 1.3 in-transit. Secrets in Vault, never in repo env files. PII pseudonymization for analytics. Annual pentest + bug bounty.

21.4 Compliance certifications target

  • Year 1: GDPR + SOC 2 Type I
  • Year 2: SOC 2 Type II + ISO 27001
  • Year 3: PCI DSS Level 2
